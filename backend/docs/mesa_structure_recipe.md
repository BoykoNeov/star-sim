# MESA interior-structure profile recipe (the `/structure` data)

The `/structure` route + `structure.py` sibling (the honest successor to the
Lane–Emden panel) serve a **real** radial structure — ρ(r), T(r), P(r),
composition(r), and the true convective/radiative boundaries — read from offline
MESA `profile.data` snapshots under `data/mesa_profiles/`. MIST ships no radial
profiles at all, and a live structure solve is out of scope (spec §2/§9), so the
profiles are generated by running MESA once and saving a handful of snapshots along
the track. The output is gitignored (like all of `data/`), so this file is the
durable record of *how it was made*.

It reuses the solar Docker recipe (`mesa_solar_recipe.md`) almost verbatim — same
image, same inlist skeleton — with **two** additions: saving `profile.data`
snapshots, and adding the convection indicator + gradients to the profile columns.
The generating run is deliberately **not** a calibrated standard solar model (no
α_MLT/Y tuning to force L=1/R=1); the `/structure` tests gate on structure that holds
regardless (a convective envelope exists, radiative core, monotone centrally-
concentrated ρ, ρ_c/T_c of the right order) — see `tests/test_structure.py`.

## 1. Run MESA in Docker (same as the solar recipe)

Image `evbauer/mesa_lean:r24.03.1.01` (MESA r24.03.1). On Windows the Docker Desktop
Linux engine must be running first (`docker info` should respond). Drive it with
`docker cp` for the files (robust across Windows drive-sharing quirks — no bind mount):

```bash
docker pull evbauer/mesa_lean:r24.03.1.01        # ~15 GB, once
docker run -d --name mesa_struct evbauer/mesa_lean:r24.03.1.01 sleep infinity

# copy the work template + BOTH default column lists
docker exec mesa_struct bash -lc '
  cp -r "$MESA_DIR/star/work" /home/docker/solar &&
  cp "$MESA_DIR/star/defaults/history_columns.list" /home/docker/solar/ &&
  cp "$MESA_DIR/star/defaults/profile_columns.list" /home/docker/solar/'
```

## 2. `inlist_project` (solar recipe + profile snapshots)

Identical to `mesa_solar_recipe.md` §2 except for the profile-saving block at the end
of `&controls`:

```fortran
&star_job
    create_pre_main_sequence_model = .true.
    pgstar_flag = .false.
    history_columns_file = 'history_columns.list'
    profile_columns_file = 'profile_columns.list'
/
&eos
/
&kap
    use_Type2_opacities = .true.
    Zbase = 0.0152
/
&controls
    initial_mass = 1.0
    initial_z = 0.0152                     ! -> [Fe/H] ~ 0.00 (project Z_sun = 0.0152)

    xa_central_lower_limit_species(1) = 'h1'   ! stop at TAMS (central H exhausted)
    xa_central_lower_limit(1) = 1d-4

    energy_eqn_option = 'dedt'
    use_gold_tolerances = .true.
    history_interval = 1

    ! --- radial-structure profile snapshots (the interior-structure feature) ---
    write_profiles_flag = .true.
    profile_interval = 25                  ! ~a dozen profiles across the MS
    max_num_profile_models = 400           ! generous cap (default 100)
/
```

## 3. `profile_columns.list` edit — the one thing that makes or breaks it

The default profile columns already output `zone, mass, logR, logT, logRho, logP,
x_mass_fraction_H, y_mass_fraction_He, z_mass_fraction_metals` — most of what the
panel needs. **Uncomment the convection indicator + Schwarzschild gradients** (the one
thing the polytrope genuinely can't fake — the panel's reason to exist):

```
   grada         ! (line ~155)  dlnT/dlnP at constant S
   gradr         ! (line ~462)  dlnT/dlnP for purely radiative transport
   mixing_type   ! (line ~514)  MESA mixing-type code; 1 = ordinary convection
```

`structure.py` treats a zone as convective when `mixing_type == 1` (ORed with the
Schwarzschild criterion `gradr > grada` as a cross-check). The bonus columns MESA
also writes by default (`pp`, `cno`, `tri_alpha` burning) are harmless if present.

## 4. Compile, run (detached), collect the snapshots

```bash
docker cp inlist_project mesa_struct:/home/docker/solar/inlist_project
docker exec mesa_struct bash -lc 'cd /home/docker/solar && ./mk > mk.log 2>&1'
docker exec -d mesa_struct bash -lc \
  'cd /home/docker/solar && ./rn > rn.log 2>&1; echo "exit=$?" > rn.done'
# wait for rn.done (1 M☉ solar to TAMS ≈ 4–5 min incl. first-run rates-cache build)
```

**Select a handful spanning ZAMS→TAMS by central H** (drop the redundant pre-MS
Hayashi profiles — all cool, fully-convective, Xc≈initial). For the 1 M☉ run, the MS
is well-sampled by the later profiles; the shipped slice keeps ~5:

```bash
# inspect each profile's model / age / center-H (header line 3, last data row col 7)
docker exec mesa_struct bash -lc 'cd LOGS; for f in profile*.data; do
  read -a hv <<< "$(sed -n 3p $f)"; xc=$(tail -1 $f | awk "{print \$7}")
  echo "$f  model=${hv[0]} age=${hv[4]} Xc=$xc"; done'

# copy the chosen snapshots to the host (gitignored data/ tree)
DEST=data/mesa_profiles/solar_1Msun; mkdir -p "$DEST"
for n in 8 9 10 11 13; do docker cp mesa_struct:/home/docker/solar/LOGS/profile$n.data "$DEST/"; done
docker rm -f mesa_struct                # done — the 19.6 GB image can stay cached
```

The `structure.py` index reads each snapshot's `(initial_mass, initial_z→[Fe/H],
star_age, center_H)` straight from the profile **header** — no `profiles.index`
needed — and snaps a request to the nearest saved snapshot in (mass, [Fe/H], age).

## 5. Measured result (the shipped 1 M☉ solar slice)

Five snapshots, near-ZAMS → TAMS:

| profile | model | age | Xc | note |
|---|---|---|---|---|
| 8  | 175 | 0.11 Gyr | 0.708 | near ZAMS (flat X, transient convective core) |
| 9  | 200 | 2.44 Gyr | 0.513 | main sequence |
| 10 | 225 | 5.09 Gyr | 0.259 | main sequence (≈ solar age → the default marker snaps here) |
| 11 | 250 | 7.12 Gyr | 0.028 | late main sequence |
| 13 | 297 | 8.37 Gyr | 1e-4  | TAMS (core H exhausted) |

Verified structure (mid-MS, profile 10): **ρ_c ≈ 190 g/cm³, T_c ≈ 1.66×10⁷ K,
R ≈ 1.06 R☉**, a **radiative core (expected_n = 3)** with a **convective envelope
from r/R ≈ 0.75 to the surface** (SSM base ≈ 0.71 R☉ — close, and not solar-
calibrated). Near-ZAMS (profile 8) shows a nearly **flat X(r) ≈ 0.71** (unburned) and
a small **real convective core** (MESA `mixing_type = 1`; the transient early-MS
¹²C→¹⁴N-burning core, before CN equilibrium), so `expected_n = 3/2` there — an honest
label flip across the snapshots, not a bug.

## 6. The 2 & 6 M☉ slice (the convective-core ↔ radiative-envelope flip) — BUILT

The natural next pedagogy is the **mirror of the Sun**: an intermediate-mass star has a
**convective core** (CNO burning, ε ∝ T¹⁶⁻¹⁸, is fiercely centrally peaked) under a
**radiative envelope** — the opposite of the 1 M☉ radiative-core/convective-envelope
case. Both 2 and 6 M☉ show it (6 M☉ most dramatically).

Same recipe, only `initial_mass` changes (`= 2.0` / `6.0`; same `Z=0.0152`, same TAMS
stop). Two operational notes learned building it:

- **Compile each work dir** (`./mk` builds `star` *into that dir*; the stock `rn` runs
  `./star`). Run them in **separate work/LOGS dirs** so the second run doesn't clobber
  the first's snapshots.
- **Snapshot selection matters more than for the Sun.** A massive-star convective core
  is largest mid-MS and *shrinks toward TAMS* (near-TAMS it can vanish — the 2 M☉ core
  is already gone by the central-H-exhausted profile). So keep at least one clearly
  **mid-MS** profile (Xc ≈ 0.4–0.5, central `mixing_type == 1`) where the core is
  healthy — don't just span ZAMS→TAMS by age or the very feature the slice exists to
  show can be missing. Inspect the central mixing type before copying:

  ```bash
  # per profile: model, age, central Xc (col 7), central mixing_type (col 15, last row)
  for f in LOGS/profile*.data; do read xc mt <<< "$(tail -1 $f | awk '{print $7,$15}')";
    echo "$f Xc=$xc centralMix=$mt"; done   # centralMix=1 -> convective core
  ```

Drop the new `profile*.data` under another `data/mesa_profiles/<run>/` dir — the
`structure.py` index globs the whole tree and snaps on the true header (mass, Z, age),
so the **runtime** needs **no code change** (the "drops in as another bucket" property).
The accompanying change is a test (`test_structure.py`, the flip locked under
`requires_structure_massive` — gated on a ≥4 M☉ slice so it *skips*, not fails, on a
1 M☉-only checkout) and this measured record — plus the one stale test whose off-grid
mass now snaps to 6 M☉ instead of 1 M☉.

### Measured result (the shipped 2 & 6 M☉ slices)

Five snapshots each, near-ZAMS → TAMS (profiles 13/15/16/18/21 for 2 M☉,
15/16/17/19/22 for 6 M☉). Central `mixing_type == 1` (a real convective core) holds
across the MS for both.

| slice | mid-MS snapshot | Xc | ρ_c | T_c | R | convective core (r/R) | envelope |
|---|---|---|---|---|---|---|---|
| 2 M☉ | profile16 | 0.39 | 71 g/cm³ | 2.3×10⁷ K | 2.07 R☉ | 0 → 0.088 | radiative (n=3/2 core) |
| 6 M☉ | profile17 | 0.41 | 16 g/cm³ | 3.0×10⁷ K | 3.89 R☉ | 0 → 0.131 | radiative (n=3/2 core) |

So `expected_n` **flips to 3/2** (the innermost zone is convective) and the envelope is
radiative at r/R = 0.9 — the exact mirror of the 1 M☉ Sun (radiative core, `expected_n
= 3`, convective envelope reaching the surface). Massive stars still carry a **razor-thin
sub-surface convection sliver** (e.g. 6 M☉ r/R ≈ 0.994–0.997, an opacity-bump zone); that
is *not* a deep envelope, so the flip test probes the bulk envelope at r/R = 0.9.
